# Key Governance — Crypto Wallet Monitoring Lab

## Overview

This document defines the operational security procedures for the
three-tier testnet wallet setup. Every decision a real institutional
custody team makes — who can sign, how backups are stored, what
triggers rotation — is documented here.

---

## Wallet Tiers

| Tier | Purpose | Signing Authority | Rotation SLA |
|------|---------|------------------|--------------|
| Hot  | Frequent operational transactions | Single authorized signer, no extra step for amounts < 0.1 ETH | 90 days |
| Warm | Intermediate / bridge tier | Manual deliberate action + written reason required | 180 days |
| Cold | Long-term storage | Full review and documented authorization before any movement | 365 days |

---

## Signing Thresholds

| Tier | Amount | Requirement Before Signing |
|------|--------|---------------------------|
| Hot  | < 0.1 ETH | Single signer — proceed |
| Hot  | 0.1–0.5 ETH | Single signer + document destination and business reason |
| Hot  | > 0.5 ETH | Treat as anomaly — verify before signing |
| Warm | Any amount | Write down: what, where, why, before signing |
| Cold | Any amount | Full review — cold wallet movement is always a significant event |

## Seed Phrase Storage

### Generation
- Generated by MetaMask on first wallet creation
- 12-word BIP-39 mnemonic

### Storage (Primary)
- Written on paper **only** — never digital, never photographed, never in a password manager
- Stored in a physical secure location

### Storage (Backup)
- Second physical copy in a **separate location**
- Geographic separation: two different rooms at minimum

### Verification (Required before relying on backup)
- After writing down the seed phrase, immediately:
  1. Open a **new, fresh browser profile** with no existing MetaMask installation
  2. Install MetaMask from scratch
  3. Select "Import wallet" and enter the written seed phrase
  4. Verify the derived wallet **address matches exactly** the original
  5. Document the verification date
- **An untested backup is not a backup.**

## Rotation Triggers

### Scheduled Rotation

| Tier | Interval | Action |
|------|----------|--------|
| Hot  | Every 90 days | Generate new wallet, migrate assets, retire old seed |
| Warm | Every 180 days | Same procedure |
| Cold | Every 365 days | Same procedure — plan carefully, cold movement is significant |

### Event-Based Rotation (Immediate)

Rotate immediately on any of the following:

| Event | Action |
|-------|--------|
| Suspected compromise | Rotate immediately, treat as incident |
| Repeated failed signing attempts (3+) | Investigate before rotating |
| Transaction to unrecognized address | Verify, then rotate if confirmed unauthorized |
| Device loss or theft | Rotate all wallets stored on that device |
| Any access by unauthorized person | Rotate all wallets |

### Rotation Procedure

1. Generate new wallet (new seed phrase, new address)
2. Test and document backup of new seed phrase
3. Move assets from old wallet to new wallet
4. Verify all assets arrived at new address on Etherscan
5. Update `config.py` with new wallet address
6. Update `APPROVED_DESTINATIONS` in config if needed
7. Destroy old seed phrase (physical shredding)
8. Document rotation: date, reason, old address (last 6 chars), new address (last 6 chars)

Only the wallet owner has access to seed phrases. No sharing.

## Incident Response

If you suspect a wallet has been compromised:

1. **Stop all signing immediately** — do not send more transactions from that wallet
2. **Document what you know** — what happened, when, how much may have moved
3. **Do not rotate yet** — preserve evidence first
4. **Check transaction history** on Etherscan for unauthorized movements
5. **Rotate the key** once you have documented the incident
6. **Review all other wallets** — if one is compromised, assume all may be

## Monitoring

The Python pipeline (`main.py` or `demo.py`) monitors for:

- Outgoing transactions to new/unknown addresses
- Single transactions above tier-specific thresholds
- Bursts of rapid outgoing transactions (drain attack pattern)
- Failed transaction attempts (unauthorized signing)

Alerts are sent to Slack with embedded response steps.
Review all alerts within the time windows defined by severity:

| Severity | Review Within |
|----------|--------------|
| CRITICAL | Immediately |
| HIGH     | Within 1 hour |
| MEDIUM   | Within 4 hours |
| LOW      | Next business day |
